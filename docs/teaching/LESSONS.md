# Lessons

This file is the course's running notebook. It's organized by lesson, not by chronology — each lesson points at the commits, PRs, files, or runbooks that demonstrate it. As the repo grows, lessons grow with it.

The lessons are deliberately *political* as well as technical. The whole premise of this repo is that data pipeline design encodes political choices — what counts as "clean," whose definitions you encode, who gets surfaced and who gets erased — and that the field rarely says so out loud.

### Working with unreliable sources

- **L04 — How to absorb format drift without breaking downstream.** [Phase 1 SEB ingestion](../../pipelines/seb_meetings/sources/youtube_rss.py) demonstrates this with the YouTube RSS feed: titles take five different shapes ("April 15, 2026", "03.18.26", "12.09.25 Part 2"), and the parser handles each as a separate, documented case. Unknown shapes return `meeting_date=None` and surface for human review rather than being silently dropped. See `_PATTERNS` for the explicit list of accepted formats. The lesson: format drift is permanent, so name every shape you accept and *leave the unknowns visible*.
- **L05 — When to fail loudly vs. quarantine vs. coerce.** The SEB pipeline does all three on purpose: (a) **Fail loudly** when YouTube changes the feed structure — `parse_feed` raises `ValueError` if `<yt:videoId>` is missing because we'd rather break and patch than silently lose data. (b) **Quarantine** unknown title formats — they make it into `seb.videos` with `meeting_date=None` and a fixable row, not the trash. (c) **Coerce** loose date strings in the Controversies sheet (`2026-04` → `2026-04-01`) because the workbook author meant "April 2026" and pretending otherwise helps nobody. The rule: fail when the data *contract* breaks, quarantine when the *content* surprises you, coerce when the *encoding* is the only thing wrong.
- **L06 — Schema contracts: writing the test you wish the upstream had.** The Pydantic `Meeting` model is the contract. It enforces enum membership on `meeting_type`, bounded floats on `hours_logged`, URL parsing on `video_url`, and `extra="forbid"` so an unexpected column raises instead of silently passing through. The warehouse mirrors the same constraints in SQL (`CHECK (hours_logged ... <= 24)`), and `test_hours_check_constraint_actually_works` proves the SQL constraint is wired — not just the Python. See [`pipelines/seb_meetings/tests/`](../../pipelines/seb_meetings/tests/). The lesson: enforce the same invariant at every layer it can be enforced at, and write a test that proves each layer enforces it. Upstream will never write that test for you.

### Politics inside pipelines

- **L07 — "Clean" is a choice. Whose definition?** Live example from the v0 workbook: meeting #1 (May 14, 2026) has a video link to `h_0CXACXv9A` — a real YouTube video, but actually the **April 22** SEB meeting, not May 14. A pipeline could (a) silently "fix" it on ingest using YouTube as a tiebreaker, (b) flag it, or (c) preserve the workbook value as-is. We chose (c), because the workbook is the human-curated system of record in Phase 1, and silently rewriting it would erase the data-quality signal. `fill_missing_video_urls_from_rss` only fills `NULL`s — never overwrites. A separate, auditable corrections workflow handles known errors with full provenance. See [`warehouse/loader.py`](../../warehouse/loader.py) `fill_missing_video_urls_from_rss`. **The lesson: "clean" is whatever your downstream readers will assume the data already is. Every implicit fix is a political act — make them explicit instead.**
- **L08 — When the qualitative columns matter more than the quantitative ones.** The `controversies`, `compliance_notes`, and `key_decisions` columns on `seb.meetings` are free-text and *not* auto-classified. The pipeline does not try to label "FBI raid" as `severity=high` or sort decisions into a taxonomy. It stores what humans wrote, with provenance, and surfaces it. The hours/dates/quorum numbers are the easy part; the *judgment* about whether a meeting was compliant, about what counts as a controversy, about which decisions reshape voting rights — that's the work, and it stays in human hands. The compliance enum has a `Unreviewed` default exactly so the warehouse doesn't lie about coverage. **The lesson: if a column requires judgment, do not let the pipeline supply it.**
- **L09 — Who can change the warehouse, and how do we record it?** A pipeline that refuses to silently rewrite the workbook (L07) needs a *sanctioned* way to correct known errors — otherwise the right move becomes "edit DuckDB by hand," which is the move you cannot audit. The corrections workflow is the answer. A correction is a YAML entry under [`corrections/`](../../corrections/) (e.g. [`seb_meetings.yaml`](../../corrections/seb_meetings.yaml)), reviewed in a normal PR under CODEOWNERS + branch protection, logged immutably in [`seb.meeting_corrections`](../../warehouse/schema/seb_corrections.sql), and applied as the final step of the ingest flow. Three invariants make it safe: (1) **append-only** — the audit table is never updated, so a revert is itself a new correction that references the prior id; (2) **whitelisted columns at two layers** — Python's `ALLOWED_COLUMNS` and SQL's `CHECK (target_column IN (...))` must be changed together; (3) **idempotent on `correction_id`** — re-running the flow is a no-op for already-applied corrections, so the ingest can run on every commit without compounding. The working example is the [May 14 video URL fix](../runbooks/corrections.md): the v0 workbook linked meeting #1 to `h_0CXACXv9A` (actually the April 22 meeting); a single correction file overrides it to `uGWdZ-DmGDA` with the SOS agenda PDF as evidence, and the audit table preserves the workbook's wrong value as `original_value` forever. **The lesson: "who is allowed to change the data" is a governance question, not a database question. Encode the answer in code review, in an append-only log, and in column allow-lists — not in a Slack message that says "please don't run UPDATEs."**
- **L09b — The schema is partly law.** Most pipelines have a schema that's an engineering artifact — chosen for normalization, performance, or developer convenience. The voter-file pipeline's schema is something else: it's a statutory promise rendered as code. [O.C.G.A. § 21-2-225(b)](https://law.justia.com/codes/georgia/title-21/chapter-2/article-6/section-21-2-225/) names exactly which fields are public and which are confidential, and [ADR-0004](../adr/0004-voter-file-sources-and-ethics.md) translates that list into invariants the model and warehouse both enforce: SSN, driver's-license number, email, month + day of birth, and registration location do not exist as columns in `pipelines/voter_file/transforms/models.Voter` or in `warehouse/schema/voter.sql`. `Voter`'s `extra="forbid"` raises if a source ever delivers a record containing them, and a parametrized test iterates `STATUTORY_CONFIDENTIAL_FIELDS` to prove every name in the statute raises on attempt. A third test reflects DuckDB's `information_schema.columns` against the same list to catch SQL-side drift. The full date of birth is treated the same way — only `birth_year` exists, because the statute makes year of birth public but month + day confidential. **The lesson: when the rules come from law, code them at the layer that makes "we forgot" impossible. A column that doesn't exist cannot leak. A model that won't construct cannot be persisted. "We promise to filter on read" is the security claim that fails first under audit — don't make it.**
- **L09c — Filter vs. rewrite: when an audit log isn't enough.** L09 covered corrections — the sanctioned way to *change a sourced value*. Suppressions look superficially similar (YAML → PR → append-only log) but they're a categorically different kind of write. A correction says: "the value in the source was wrong, here's the right one, log both." A suppression says: "the value in the source is correct and we have no public mandate to change it, but a specific person asked us to leave them out of our public outputs." The mistake to avoid is solving the second problem with the first tool. Imagine the suppressions workflow as written: YAML, PR, append-only log, and — fatally — an UPDATE that nulls out the voter's row, or sets a `suppressed=true` flag. Now the public query that forgets to filter on `suppressed=false` quietly exposes data we promised wouldn't be there, the operator who runs `SELECT * FROM voter.voters` for a one-off task sees the suppression baked into the data, and reversing a suppression means resurrecting values from the audit table. So this pipeline does it differently. [`voter.suppressions`](../../warehouse/schema/voter.sql) is the audit log; [`voter.active_suppressions`](../../warehouse/schema/voter.sql) is a view computing "currently-effective suppress" via `LAST(action ORDER BY requested_at)`; [`voter.public_voters`](../../warehouse/schema/voter.sql) is the *canonical safe surface* every public output reads from. The underlying voter row is byte-identical before and after a suppress — there's a parametrized test for exactly that. To reverse a suppression you append an `unsuppress` row with `supersedes` pointing at the prior id; the view recomputes and the voter re-appears in `public_voters` automatically. **The lesson: when an audit log records intent, decide whether it should also *enforce* that intent by rewriting data — or whether the data should stay intact and the filter should live in a view. A rewrite is an answer to "what does the warehouse hold?"; a filter is an answer to "what does the public see?" Confusing the two produces an apologetic incident report.**
- **L09d — The cross-pipeline boundary is a join shape, not a join key.** ADR-0001 named cross-pipeline analytics as the whole reason both pipelines live in one repo: "what did the board decide, and what changed on the ground after." The seductive way to answer that question is to find a column the two pipelines share — a county field on SEB meetings, a topic tag on voter records, anything — and join on it. The right answer is to refuse. SEB meetings have free-text `key_decisions` and `controversies` columns; the pipeline does not auto-classify which county a specific decision affected, because doing so would be the pipeline supplying judgment (see L08). And the voter side, per [ADR-0004 Rule 4](../adr/0004-voter-file-sources-and-ethics.md), is forbidden from exposing any column that could re-identify a voter when combined with another dataset. So the join cannot be on content, and it cannot be on per-voter identifiers. What's left is *shape*: bin SEB activity by calendar quarter, roll voters up to county, and cross-join the two. The result is a long table where every (year, quarter, county) cell shows SEB compliance-status counts beside that county's registration shape — enough for a researcher to ask "in the quarter the SEB flagged something, what did the county look like?" and *not* enough for the pipeline to claim causation. Three rules fall out: (1) cross-pipeline files live in [`warehouse/queries/`](../../warehouse/queries/), never in `warehouse/schema/`, because a schema file describes *one* pipeline; (2) the cross-pipeline view composes per-pipeline aggregate views, never base tables — [`analytics.seb_voter_overlap`](../../warehouse/queries/seb_voter_overlap.sql) reads `voter.county_registration_summary` which reads `voter.public_voters`, so suppressions cascade automatically; (3) the column list of the cross-pipeline view is itself a privacy contract — [`test_overlap_view_excludes_per_voter_identifiers`](../../pipelines/voter_file/tests/test_analytics.py) hard-codes a forbidden-column list so a future change that adds `precinct` or `zip5` fails CI loudly. **The lesson: the question "what did decision X do to county Y" is a researcher's question, not a pipeline's question. The pipeline's job is to put the two sides next to each other with honest temporal and geographic shape, and to refuse the joins that look convenient but would either fabricate causation or leak the people the statute promised to protect.**
- **L09e — Distribution ethics: what we publish vs. what we hold.** The warehouse holds more than the public API serves. `voter.voters` has per-voter rows; `voter.suppressions` records who asked to be filtered; `voter.precinct_registration_summary` rolls up to a unit small enough to re-identify against an external dataset. None of those reach the public surface, and the way none of them reach it is the lesson. Two layers of enforcement, separated on purpose so a single mistake at one layer cannot leak. **Layer 1 is the SQL view chain.** `voter.public_voters` is a view over `voter.voters` that filters out anyone whose most recent suppression action is `suppress`. Every aggregate view (`voter.county_registration_summary`, `analytics.seb_voter_overlap`) reads from `voter.public_voters` or downstream of it. Suppressions cascade automatically; a per-voter row never enters a public aggregate. **Layer 2 is the Python allow-list.** [`outputs/api/_allowed_sources.py`](../../outputs/api/_allowed_sources.py) names every `schema.name` the API is permitted to query. [`tests/test_api_allowed_sources.py`](../../outputs/api/tests/test_api_allowed_sources.py) walks every handler in `outputs/api/routes/` and `bulk_export.py`, regexes out every `FROM` clause, and asserts each one is on the allow-list. A second parametrized test asserts no handler queries any name on `EXPLICITLY_DENIED_SOURCES` (`voter.voters`, `voter.suppressions`, `voter.active_suppressions`, `voter.precinct_registration_summary`, `seb.meeting_corrections`). The denial list is redundant with the allow-list by construction — that's the point. A contributor who accidentally writes `FROM voter.voters` fails two tests with two different messages, one mechanical, one reasoned. Layered enforcement makes one-line mistakes survivable. **The third decision is the bulk-versus-API split.** Researchers who want the full `voter.county_registration_summary` table do not get it from the paginated API; they get it from a separately-published CSV/Parquet artifact regenerated per warehouse build, with `sha256` per file in `MANIFEST.json`. The split is not just performance; it's an act of distinction. The API is for interactive reads. The bulk artifact is a publication event, versioned and hash-pinned, that says: "we shipped this, with these columns, at this time, deliberately." An adversary who wants everything can have everything we already publish, but the act of taking it is a recorded download against a named file, not a sustained iteration over a paginated endpoint. **The fourth decision is the rate limit and the log format.** 100 requests per minute per IP is generous enough that asking permission would itself be a barrier; the limit exists so sustained extraction is visible in logs, not so reads are gated. The log format is part of the ethics: timestamp, route, status, latency, response bytes, and a per-day hashed IP. Never the IP. Never query parameters. The salt rotates at UTC midnight so we can answer "did this source make 50,000 requests in an hour" without ever answering "did this person come back tomorrow." Observability-grade auditing, not surveillance-grade. **The lesson: distribution is the moment your invariants get tested in public. If your privacy contract is one filter or one route decorator, the first bug evicts it. If it is a chain of views, an allow-list, a denial list, two tests, a paginated read path, a separately-versioned bulk path, and a log format that cannot record the thing it must not record, then no single mistake is sufficient to break the contract.** See [ADR-0005](../adr/0005-public-read-api.md).

### Testing

- **L13 — Property tests over fixture tests where the universe is bigger than your examples.** Fixture tests answer "does this exact input produce this exact output?" That's the right tool when the input space is small or the cases are adversarial — the [`test_bulk_file.py`](../../pipelines/voter_file/tests/test_bulk_file.py) parametrization over [`STATUTORY_CONFIDENTIAL_FIELDS`](../../pipelines/voter_file/transforms/models.py) is a fixture test, and it should be: every name in the list matters, none of them can drift, and the test reads like a contract. But [`transforms/address.py`](../../pipelines/voter_file/transforms/address.py) composes display strings from six free-form fields (house number, street name, suffix, apartment, city, ZIP). The input universe is effectively infinite. The invariants are easy to state — round-trip on the composed surface, idempotence of normalization helpers, refusal to silently emit incoherent output — but hand-picking fixtures to cover them is a fool's errand: the fixture you forget is the bug your users find. [Hypothesis](https://hypothesis.readthedocs.io/) lets you state the invariant once (`compose ∘ split == identity` on full parts) and search the space for a counterexample. When it finds one, it shrinks to the smallest input that breaks the property and reports that, so the failure message is a minimal reproducer, not a haystack. The trick is scoping: don't reach for Hypothesis when the cases are adversarial (use parametrize), don't reach for it when you only have three examples (just write three tests), and don't widen the strategy until you're testing Unicode normalization in a module that's never going to see Unicode. **The lesson: pick the test style by the shape of the input space, not by reflex. Small, adversarial, named cases get fixtures. Large, structured, well-invariant spaces get properties. The two styles answer different questions — "is this exact case handled?" vs. "does this invariant hold?" — and a real test suite needs both.**
- **L14 — Contract tests: protecting downstream from upstream.** `test_missing_required_element_raises` and `test_malformed_feed_raises_loudly` (in `test_youtube_rss.py`) lock in the assumption that the YouTube Atom feed contains `<yt:videoId>` on every entry. If YouTube ever drops that field, those tests fail in CI before the ingest does in production. The fixture itself is a captured payload — the contract is "this is what we believed the upstream was" and the test is "if that belief breaks, the build fails." **The lesson: pin the upstream's behavior with a test, not a comment.**
- **L15 — Golden-file tests for transforms that aren't worth re-deriving in test code.** `test_workbook_seed.py` runs against `fixtures/seb_meetings_v0.xlsx` rather than a hand-built Python dict. The workbook *is* the input the pipeline ingests in production — reproducing it in code would be slower, more fragile, and would test only what we remembered to write. Tests assert concrete row counts (`17 meetings, 8 controversies, 15 sources`) so any silent drift surfaces. **The lesson: when the production input is a file, make the file your fixture; let the file's shape be the test.**
